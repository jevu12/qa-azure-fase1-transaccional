Skill: QA Azure Fase 1 Transaccional

Ejecuta la Fase 1 transaccional de QA en Azure DevOps: deteccion, creacion idempotente de QA Tasks, Test Plan/Suite/Test Cases y vinculos, con trazabilidad en decisions_log y respeto estricto de estados permitidos.

🎯 Cuando usar esta skill?

Usa esta skill cuando el usuario pida operar QA directamente en Azure DevOps para:

  • Detectar artefactos existentes por User Story
  • Crear solo artefactos faltantes (QA Tasks, suite, casos)
  • Enlazar US ↔ Task ↔ Test Case
  • Dejar trazabilidad auditable de decisiones
No uses esta skill para generacion de codigo, PR automation, Appium o clasificacion avanzada de fallos (Fases 2-4).

📥 Entradas esperadas

ParametroDescripcionRequerido
project Nombre del proyecto Azure DevOps Si
iteration_path Sprint completo (ruta de iteracion) Si
mode full-pipeline · analysis-only · design-only · execution-only · detection-only Si
user_stories Lista de IDs de US; vacio = todas del sprint Opcional

Fuente de configuracion obligatoria: inputs/project-config.json.

Guia general: references/guia-configuracion-general.md. Templates: templates/project-config.template.json y templates/.env.template.

🎳 Modos de ejecucion

ModoDescripcionEstado
full-pipeline Deteccion + ingesta + analisis + diseno completo Disponible
analysis-only Solo deteccion + ingesta + analisis QA Disponible
design-only Solo deteccion + ingesta + diseno de Test Cases Disponible
execution-only Ejecucion/reporter/evidencias — requiere autorizacion del Orquestador Extension compat
detection-only Solo deteccion y diagnostico, sin mutaciones Disponible

⚙️ Proceso obligatorio

El siguiente orden de pasos es estricto y no puede alterarse:

01

Cargar y validar configuracion

  • Leer inputs/project-config.json
  • Validar: organization.url, project.name, project.team, project.iteration_path, execution.mode
  • Resolver sprint_full_path desde project.iteration_path
  • Si falta config: bloquear (CONFIG_INVALID), salvo detection-only con fallback a inputs/us-list.md
02

Verificar MCP Azure DevOps (obligatorio)

  • Confirmar disponibilidad MCP con lectura simple de proyectos
  • Confirmar acceso a project.name, team e iteracion objetivo
  • Resolver usuario autenticado MCP para reglas de asignacion/ownership
  • Si falla validacion MCP: detener mutaciones y devolver BLOCK con diagnostico
03

QA Orchestrator Service (obligatorio)

  • Consultar estado actual de la User Story
  • Validar si la historia es procesable segun las reglas del flujo
  • Verificar tareas QA, casos de prueba, suites, ejecuciones, bugs, comentarios y evidencias existentes
  • Aplicar reglas de idempotencia, deduplicacion y gobernanza
  • Seleccionar el agente adecuado y el foco de ejecucion
04

Cargar estado compartido

  • Activar lock de concurrencia para outputs/pipeline-state.json
  • Si no existe, inicializar segun contrato de references/contratos.md
  • Si existe pero es de otro proyecto/sprint, iniciar nueva ejecucion
05

Obtener US objetivo

  • Si execution.user_stories trae IDs, usar esa lista
  • Si viene vacio, consultar US del sprint en Azure DevOps
06

Deteccion por US (siempre antes de crear)

  • Leer US completa con relaciones
  • Detectar: QA Tasks hijas, Test Cases vinculados, Suite/Plan relacionados, comentarios QA, evidencias
  • Registrar hallazgos en stages.deteccion y decisions_log
07

Validaciones de elegibilidad

  • Estado permitido: aplicar reglas de references/reglas-transiciones.md y references/reglas-decision.md
  • Contenido minimo: description y criterios de aceptacion
  • Filtro tecnico: execution.exclude_technical_keywords
08

Acciones transaccionales de Fase 1

  • Crear QA Tasks faltantes (analisis, diseno, ejecucion) como hijas de la US
  • Crear/ubicar Test Plan y Test Suite del sprint
  • Crear Test Cases faltantes
  • Vincular Test Cases a US (Tests/Tested By) y a QA Task de diseno (Related)
  • Completar artefactos parciales; nunca recrear artefactos completos
09

Flujo generico execute_and_publish (si aplica ejecucion)

  • Descubrir contexto por historia (planId, suiteId, testCaseIds)
  • Resolver Test Points y construir mapa testCaseId → testPointId/revision/title
  • Validar cobertura; si falta test point: bloquear setup (BLOCKED_SETUP)
  • Crear run planificado; guardar runId
  • Publicar resultados por lote con outcome, state=Completed, comment
  • Gestionar evidencias bajo outputs/evidencias/<sprint>/US-<us_id>/TC-<tc_id>/
  • Validar cobertura por paso; si falta evidencia: bloquear cierre (BLOCKED_EVIDENCE)
  • Cerrar run solo con evidencia validada y registrar trazabilidad en pipeline-state
10

Persistencia y auditoria

  • Actualizar outputs/pipeline-state.json por etapa
  • Registrar toda decision en decisions_log con motivo, codigos y artefactos afectados

📐 Reglas obligatorias

Idempotencia primero: nunca crear sin detectar antes
Anti-duplicado por titulo, patron y relacion de negocio
Sin operaciones destructivas ni eliminacion de artefactos
Sin mutaciones si MCP no esta validado
PAT solo desde AZURE_DEVOPS_EXT_PAT; sin hardcodeo ni logging
Evidencias segmentadas por US/TC, nunca a nivel US
No cerrar run sin evidencia validada por paso (BLOCKED_EVIDENCE)
Identidad resuelta desde usuario autenticado MCP, sin IDs hardcodeados
Mantener AreaPath e IterationPath alineados con la US
Si falta info critica de testabilidad: next_action = WAIT_USER_INPUT
Ningun agente ejecuta en ADO sin autorizacion previa del Orquestador
Bugs: asignar al ultimo asignado de la US distinto al QA autenticado MCP

  • Generacion de codigo de automatizacion
  • Creacion/revision automatica de Pull Requests

  • Clasificacion avanzada de fallos o creacion inteligente de bugs multi-causal (Fases 2-4)

markdown · Resumen Fase 1

## Contexto
- Proyecto: <project>
- Sprint: <iteration_path>
- Modo: <mode>
- US procesadas: <n>

## Resultado por User Story
- US #<id> (<state>): <CREATE|UPDATE|SKIP|BLOCK|ERROR>
- Detectado: <tasks/suite/tcs existentes>
- Accion: <que se creo o completo>
- Razon: <por que>
- Artefactos afectados: <IDs>

## Trazabilidad global
- QA Tasks creadas: <n>
- Suites creadas: <n>
- Test Cases creados: <n>
- Vinculos creados: <n>
- US omitidas/bloqueadas: <n>
- Archivo actualizado: outputs/pipeline-state.json

✅ Checklist rapido de calidad

  • Se leyo project-config.json antes de decidir?
  • Se ejecuto deteccion previa a toda creacion?
  • Se evito duplicacion por estado y relaciones?
  • Se registraron decisiones en decisions_log?
  • La evidencia quedo segmentada por US/TC y validada por paso antes del cierre?
  • Se respeto el limite de alcance Fase 1?
